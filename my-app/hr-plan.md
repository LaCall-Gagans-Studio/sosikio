# /hr 抜本改編プラン（重厚化 + デモ導線 + リード獲得）

## 1. 目的とゴール

- 現在の `/hr` の世界観（黒 × 蛍光イエロー × マゼンタ、ハザード演出）は維持しつつ、意思決定者向けの本格LPにアップグレードする
- 「読むLP」から「体験して判断できるLP」へ進化させ、デモ体験経由で高意欲リードを獲得する
- フォーム完了率を上げるため、段階的入力（1項目ずつ + 進捗表示）で離脱を抑える

KPI（初期想定）:
- デモ開始率（`demo_open / lp_view`）: 20%以上
- 録音完了率（`record_stop / demo_open`）: 60%以上
- リード完了率（`lead_complete / demo_open`）: 25%以上
- 送信完了CPA改善（既存LP比）: -20%

## 2. 体験設計（情報設計 + 導線）

### 2-1. ページ構成（改編後）

1. **グローバルヘッダー（新規）**
   - 左: SOSIKIOロゴ（`/`へリンク）
   - 右: `Probe` `HR特設` `理念・会社情報` `資料請求`
   - スクロール時に半透明 + ブラーで固定

2. **Hero（強化）**
   - 現行コピーを基盤に、導入判断に必要な実務メリット（予兆検知、離職抑制、既存施策との併用）を追加
   - CTAを2系統:
     - 「デモを体験する（右下導線と同一挙動）」
     - 「資料請求」

3. **課題提起（既存の3ケース拡張）**
   - ケースごとに「見逃した理由」「検知できるサイン」「取るべき介入」を追加
   - 読み物としての深度を上げる

4. **診断ロジック（主観×感情）**
   - 既存説明を再構成し、入力負荷/精度/運用フロー（誰が・いつ・どう使う）を明文化

5. **ケースグラフ（既存活用）**
   - 現行グラフを維持しつつ、解釈ガイド（管理職が次に何をすべきか）を追記

6. **導入ステップ（新規）**
   - 30日導入ロードマップ
   - 初期設定、運用定着、レポート活用までの時系列提示

7. **信頼性セクション（強化）**
   - 金間メソッド、研修、書籍、組織への適用指針を再編成

8. **FAQ（新規）**
   - 精度、個人情報、録音運用、導入負荷、既存サーベイ併用などの意思決定阻害要因を先回りで解消

9. **フォームセクション（既存 + 改修）**
   - デモ結果受け取り導線と同じトーンで統一

10. **グローバルフッター（新規）**
    - サイト導線、法務リンク、会社情報、`/`リンク
    - `/probe` `/hr` への導線明示

## 3. 常時右下デモ導線（新規）

### 3-1. 固定FAB（Floating CTA）

- 常時右下に固定表示（モバイル/PC共通）
- 文言例: `最短3秒で音声感情分析`
- クリックで **/hr/demo** を開く
- スクロール深度やセクション到達に応じてマイクロアニメで注意喚起

### 3-2. デモ体験フロー（モック + 実マイク許可）

1. **録音開始画面**
   - マイク許可要求（`getUserMedia`）
   - 許可後、録音ボタン活性化
   - 録音中は波形（Web Audio API で可視化）を動かし「記録している感」を演出
   - 実データ保存は不要（モック）

2. **録音停止**
   - 停止後、擬似解析画面へ
   - 2～4秒程度の解析演出（進捗バー + ステータス文言）

3. **解析結果（固定データ）**
   - 上部に簡易結果（例: 活力度、ストレス傾向、発話安定度）
   - 下部に詳細結果エリアを表示するが、モザイク + ロック表示

4. **詳細閲覧ゲート（段階フォーム）**
   - 「結果を見る」クリックで段階入力開始
   - 入力項目:
     - Step1: メールアドレス
     - Step2: 会社名
     - Step3: お名前
   - 各ステップ1項目のみ表示
   - 進捗バー（33% / 66% / 100%）
   - 入力途中保存（state保持）

5. **完了画面**
   - 文言: `結果がメールで届くのをお待ちください`
   - 追加CTA: `担当者に相談する`

## 4. リード獲得実装設計

## 4-1. 取得データ

- 必須: `email`, `company`, `name`
- 任意: `demo_started_at`, `record_duration_sec`, `diagnosis_variant`, `page_path`
- 付与: `source='hr-demo-gate'`, `utm_*`, `referrer`

## 4-2. 保存先 / 通知方式

- **既存のお問い合わせと同じ方式**で実装する
  - `POST /api/contact` に `type: 'hr-demo'` を追加
  - サーバー側（nodemailer）で管理者宛にメール送信
- 送信メール本文に、以下を含める
  - `email`, `company`, `name`
  - `record_duration_sec`, `diagnosis_variant`, `source`, `page_path`
  - `utm_*`, `referrer`（取得できる範囲）
- 管理運用は「問い合わせメール受信箱」で一元化し、既存フローに乗せる

## 4-3. 送信タイミング

- Step3完了時に API 送信
- 失敗時はリトライUI
- 成功後は完了画面へ遷移

## 6. UI/UXの実装方針

- デモ体験は **モーダルではなく `/hr/demo` の独立ページ**として実装
- 右下FABやHero CTAから `/hr/demo` へ遷移
- `/hr/demo` は「録音 → 解析 → 結果ゲート」の1フロー専用レイアウトにする
- モバイルでの安全領域（`env(safe-area-inset-bottom)`）考慮
- アクセシビリティ:
  - フォーム要素のラベル明示、見出し階層を維持
  - ボタン/入力のラベル明示
  - マイク拒否時の代替文言と再試行導線

## 9. リスクと対策

- **マイク拒否率が高い**
  - 拒否時の代替体験（サンプル音声で体験）を用意
- **入力離脱**
  - 1項目ずつ + 進捗 + 完了まで短い導線
- **演出過多で重い**
  - 波形はCanvas軽量実装、解析は固定アニメのみ
- **法務/プライバシー**
  - 録音データは保存しない旨を明示
